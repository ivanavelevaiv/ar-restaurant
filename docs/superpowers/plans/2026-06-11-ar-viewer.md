# AR Viewer Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the placeholder ar-viewer.html with a working marker-based AR scene using A-Frame v1.3.0 + AR.js v3.4.5, with slug-based dish routing, a loading indicator, and a "coming soon" fallback for unmapped dishes.

**Architecture:** The page always loads A-Frame/AR.js but only injects `<a-scene>` into the DOM when the URL slug maps to a known model — this avoids triggering camera permission prompts for dishes without models. All UI chrome (back button, instruction strip, overlays) lives in `position: fixed` divs above the z=0 A-Frame canvas.

**Tech Stack:** A-Frame v1.3.0, AR.js v3.4.5 (A-Frame/marker build), vanilla JS, static HTML/CSS (no build step)

        // Add new entries here as models become available.
        // Key   : the ?dish= slug used in the URL

                    </a-assets>

                        <!-- 3D dish model — see TWEAK constants above -->
                        <a-entity
                            id="dish-entity"
                            gltf-model="#pasta-model"
                            scale="${SCALE}"
                            position="${POSITION}"
                            rotation="${ROTATION}">
                        </a-entity>

                        <!-- Ambient light: lifts shadows on PBR textures -->
                        <a-light type="ambient"
                                 color="#ffffff"
                                 intensity="0.6">
                        </a-light>

                        <!-- Directional light: primary PBR light source -->
                        <a-light type="directional"
                                 color="#fff9e6"
                                 intensity="1.2"
                                 position="1 2 1">
                        </a-light>

                    </a-marker>

                    <a-entity camera></a-entity>
                </a-scene>`;

            const scene = template.content.firstElementChild;

            // Hide loading overlay once glTF model finishes loading
            let loaderDismissed = false;
            function hideLoader() {
                if (loaderDismissed) return;
                loaderDismissed = true;
                const overlay = document.getElementById('loading-overlay');
                overlay.classList.add('fade-out');
                setTimeout(() => { overlay.style.display = 'none'; }, 450);
            }

            scene.addEventListener('loaded', () => {
                const entity = document.getElementById('dish-entity');
                if (entity) {
                    entity.addEventListener('model-loaded', hideLoader);
                }
                // Fallback: dismiss loader after 10 s if the event never fires
                // (e.g. model already cached, or slow network timeout)
                setTimeout(hideLoader, 10000);
            });

            document.body.appendChild(scene);
        }
    </script>

</body>
</html>

**Date:** 2026-06-11

### Summary
Replaced the static placeholder in `ar-viewer.html` with a fully functional marker-based AR scene. The viewer loads a `.glb` model on top of a printed Hiro marker using the device camera. | Library | Version | CDN |
|---|---|---|

| AR.js (A-Frame build) | 3.4.5 | `cdn.jsdelivr.net/gh/AR-js-org/AR.js@3.4.5` |

### Architecture Decisions

**Scene injection, not static markup:** The `<a-scene>` element is built as a DOM template and appended to `<body>` only when the slug resolves to a known model. This prevents AR.js from requesting camera access for dishes that have no 3D model yet.

**Slug-based routing:** The `?dish=` query parameter now uses human-readable slugs (`shrimp-pasta`) instead of numeric indices. The `MODEL_MAP` object in `ar-viewer.html` is the single source of truth — add new entries there as models are ready.

**Z-index layer stack:**
- z=0: A-Frame canvas (webcam + 3D render)
- z=10: Instruction strip (semi-transparent top bar)
- z=20: Back button + nav logo (permanent chrome)
- z=30: Loading overlay / coming-soon overlay (topmost)

**Loading indicator:** A spinner overlay (`#loading-overlay`) is shown immediately on page load and dismissed via the `model-loaded` A-Frame entity event. A 10-second fallback timeout prevents the spinner from hanging if the event does not fire.

**PBR lighting:** Two lights are placed inside `<a-marker>` so they track with the marker reference frame — an ambient fill (`intensity: 0.6`) and a warm directional key light (`intensity: 1.2`, position `1 2 1`). `renderer="colorManagement: true"` is set on the scene to ensure physically correct tone-mapping.

### Model Attributes (tweak after first physical test)
| Attribute | Value | Notes |
|---|---|---|
| scale | `0.2 0.2 0.2` | Conservative start; increase if model appears small |
| position | `0 0.05 0` | Slight Y lift to sit model base on marker surface |
| rotation | `0 0 0` | Try `-90 0 0` if model renders flat/horizontal |

### iOS / Android Notes
- Camera access requires **HTTPS** in production (getUserMedia restriction).
- `apple-mobile-web-app-capable` meta tag included for iOS Safari viewport behaviour.
- Testing on local network: serve over HTTP (`python -m http.server 8080`); iOS Safari on the same Wi-Fi network will prompt for camera via HTTP on localhost-equivalent IPs.
